# 用毕业论文理解 Git

## 传统做法的问题

我们写毕业论文时，文件常常变成这样：

- 毕业论文_初稿.docx
- 毕业论文_修改版.docx
- 毕业论文_导师意见版.docx
- 毕业论文_最终版.docx
- 毕业论文_最终版2.docx
- 毕业论文_真的最终版.docx
- 毕业论文_这次真的是最终版.docx

问题在哪：

- 不知道每一版到底改了什么，只能靠文件名瞎猜
- 导师给的意见混在一堆"最终版"里看不见
- 两个同门同时帮你改，最后不知道哪一版才是合并完的
- 想回到"两周前那一版"基本是噩梦

## Git 怎么解决

把上面的混乱翻译成 Git 的语言：

| 传统做法 | Git 做法 |
| --- | --- |
| 另存为"最终版2" | 一次 commit，带说明，记录谁什么时候改了什么 |
| 单独开一版改第三章 | 新建一个 branch，不影响正式稿 |
| 邮件发给导师审 | 发一个 PR，让导师在线 review，留下评论 |
| 导师说"按你最新版改" | merge 进 main 分支，正式稿就更新了 |
| 两个同门改了同一段 | Git 报 conflict，让人来决定保留哪边 |

## 一个具体场景：导师让改研究背景

1. 从 main 新建分支 feature/intro-revision
2. 改 introduction.md，第一次 commit："重写研究背景第二段"
3. 改完发现引用格式不对，第二次 commit："修正引用格式"
4. push，发 PR，标题写"重写研究背景"
5. 导师 review，留评论："这段需要补一篇 2023 的文献"
6. 你按意见再加一个 commit："补充 2023 文献"
7. 导师 approve，merge 进 main
8. 一个月后想看"当初导师让我加的是哪一句"，直接看那次 commit 的 diff 就行

整个过程没有"最终版_真的最终版.docx"，只有清晰的修改历史。

## 导师意见如何追踪

传统做法：导师在 Word 里用批注，你改完关掉批注，下次哪些改了哪些没改谁也说不清。

Git 做法：

- 导师在 PR 的 Files changed 里点行号加 inline comment
- 评论保留在 PR 页面，永远可查
- 你的每一次回应都是一个新 commit，commit message 写清楚"按导师意见 X 修改"
- 答辩前回看：打开 PR，所有意见和你的回应一目了然，不用翻邮件

这就是为什么 Git 协作不仅是"管文件"，更是"管协作记录"。
